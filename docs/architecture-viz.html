Architecture Principles

A2A Protocol
Each agent is an independent FastAPI service. API dispatcher reads routing state and sends JSON-RPC 2.0 tasks/send.
Routing Table
Dispatcher is a pure lookup: (phase, active_npc_id) → agent URL. No business logic in the gateway.
MCP Tool Protocol
Agents call data stores via HTTP MCP servers. X-Campaign-ID header injected transparently — no manual filtering.
4 Provider Services
LLM, image, TTS, and STT are separate adapters. Swap any backend (Gemini↔Anthropic, ElevenLabs↔OpenAI) without touching agents.
Memory-Agent Bridge
DM and NPC agents never touch memory-mcp directly. memory-agent owns STM/LTM compression and returns consolidated context.
Campaign Isolation
Every Postgres row, Qdrant point, and Neo4j node carries campaign_id. All indexes are composite — no cross-campaign leakage possible.

NPC Multi-Turn Conversations

NPC conversations are genuine multi-turn dialogues — the player and NPC go back and forth until one of them says goodbye. The DM initiates and closes; the api dispatcher handles all in-between turns.

NPC Context Layers

Each npc-agent turn rebuilds context from three distinct layers with clear role labels:

Layer 1 — Preamble (3 turns before conv_start)
Gives the NPC continuity: why did the player approach? What did the DM just narrate?

[DM] "You push open the tavern door..."
[Player] "I approach the innkeeper."
[DM] "She sets down her cloth and turns."
Layer 2 — DM Briefing (dynamic, per-conversation)
Generated by the DM when starting the NPC conv. Tells the NPC what to do, not who to be (that's the static profile).

goals: Protect regulars, be wary of strangers
knows: Merchants left 3 nights ago toward Loppio
mood: Tired, suspicious
reveal_if: Player seems trustworthy or offers coin
Layer 3 — Conversation so far (since conv_start)
All turns since conv_start_turn_id — player and NPC interleaved with clear labels.

[You] "Evening. What'll it be?"
[Player] "I'm looking for missing merchants."
[You] "Aye, they left three nights past..."
[Player] "Do you know where they went?"

Conversation Close Detection

How done=true works
The LLM sees the full conversation and the latest player message. The system prompt instructs: set done=true ONLY when both sides have reached a natural close — farewell words, resolved topic, or the player explicitly ending it. Examples: "Thanks, safe travels", "Goodbye", "That's all I needed", "Pleasure doing business". The model is explicitly told never to end mid-conversation. If the player sends an abrupt close ("ok bye"), the NPC responds naturally in character ("Fare well, traveller") and then returns done=true.

After the conversation — DM context

The NPC conv summary is logged as a role=system turn with metadata.type="npc_conv_summary". The DM's next get_turns(exclude_roles=["npc"]) call returns this summary turn naturally — no special injection needed. The DM sees something like:

[system] The player learned from Elara the innkeeper that the missing merchants departed three nights ago heading east toward Loppio. She seemed relieved when they left.
